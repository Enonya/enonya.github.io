    0x00. 做题思路ψ(｀∇´)ψ

    0x01. 阅读题面ψ(｀∇´)ψ

    拿到题首先应该做的是阅读题面，先形式化题面（很多时候形式化就代表了初步转化）。

    想清楚题目要求你做什么，这类问题有没有见过，有没有套路，如果见过类似的，这个题应该有什么变化？比如做过 Permutation Distance 之后再看到 230111C-T1-小W与身高 这样。

    看清楚问题是 online 还是 offline？是 exists 还是 forall？

    没有套路的话是不是应该考虑转化/观察结论？（这个下面再说）

    当然，如果一个问题思考的时候脑袋里没有一种“清晰”的感觉，就尽量不要开始下一步，比如 230203C-T1-字符串。

    观察数据范围也是一个很重要的点，可以在很大程度上快速判断对应 subtask 的时间复杂度以确定方向，比如 Union of Two Sets 这类交互，能通过交互题出题的局限性，根据数据范围快速判断做法（（（

    不过有的时候也不能被这个局限了：比如230201C-T4-回文串，发现这个东西字符集是 \(26\)，而且字符串长度 \(1e5\)，以为复杂度是 \(26 \times 1e5\)，结果实际上还要再多一个二分的 \(\log\)。

    还有 230201C-T2-翻转，这个也是被数据范围直接骗了，方向直接错完，本来就是暴力，想成状压 dp 了……

    所以观察数据范围来判定做法是下下策，只能在完全没有思路的情况下使用！

    不过你为了确定做法值不值得想下去，还是要看一下数据范围的，避免像 USCAO23JAN Bronze T2 这样，本来就是个语法暴力题，就因为把 \(10\) 看成了 \(1e5\)，想了大半天贪心，成为小丑。

    0x02. 转化问题以及观察结论和性质ψ(｀∇´)ψ

    这个其实才是很多时候打开题目突破口的地方。

    有一些比较常用的技巧，在这里列举一下：

    • 最笨的方法就是考虑手玩样例，模拟一下题面的过程看看有没有启发，找找规律试试？比如：220518C-T2-「客星璀璨之夜」 这样？或者是类似 2301113C-T4-coin 这种考虑把有联系的东西放在一起，观察到问题的本质？
    • 再进一步就是考虑观察答案的形式，合法解的形状，看看有没有什么启发，比如 Young Maids，考虑一个答案合法后对原问题决策的限制是什么？还是说比如 221025C-T4-罪⼈挽歌
    • 要想清楚，什么东西才会造成贡献，一个位置什么时候合法，什么时候不合法，我们希望能类似拆开限制一样，让各个元素做的贡献尽量的减少对其它元素贡献的影响，这样更方便计算，比如类似 230203C-T4-交换 的想法，很多时候也能类似下面 dp 里提到的 Xor-Subsequence (easy version) 一样用来排除不必要的状态。
    • （接上一条）也可以考虑一下，如果我们考虑全局 or 每一段，本身就已经能满足答案的贡献，然后再考虑凑出来之后又出现的新的贡献（前提是开始的时候满足贡献的，拼凑起来之后也能满足贡献）：NOIOnline2022TG-丹钓战，CF1801C-Music Festival
    • 对于一个最优化问题，思考一下，怎么样才能更加优秀，是不是要考虑贪心啊，dp啊之类的做法了？比如 230203C-T3-序列
    • 观察一个看起来啥都没有的问题，要考虑自己手动加上一些限制，写出“不是，就是，一定是，否则”这样的观察，然后考虑这些东西有啥用：230117C-T4-子图。

    这些是用于转化问题的，或者说已经接近直接的做法了：

    • 有多个复合的限制条件的时候，考虑把限制条件拆开维护：Permutation Distance。
    • 考虑更换贡献的计算方式，选择不那么直观的贡献拆开维护？类似 Manacher 这样，只考虑以一个位置为中心的所有回文串，进一步说这个是钦定一个类似 dp 中“阶段”的基底，找到一个计算贡献的基准。
    • 或者考虑累加影响，批量处理贡献？比如 230201C-T4-回文串，计算答案的时候就是反过来，考虑基准对于所有能产生贡献的点累加一个权值，最后离线在所有点算一下权值即可
    • 也可以对于一个复杂的 \(\sum\sum\min\max\) 这样的式子，我们尝试直接钦定出单个位置的答案，求和之后再推式子ABC290F - Maximum Diameter，不然就是类似上面那样更换计算方式。
    • 是否要考虑分离变量？比如树状数组维护矩阵改矩阵和的时候，比如线段树维护方差的时候。
    • 考虑一下修改了一个位置之后，对答案的影响是什么？这里还是以 230201C-T4-回文串 为例，这个题确实比较深刻。
    • 考虑一下，正着做很麻烦，能否考虑反着做，或者拆开做？
    • 当前的式子，当前的一些变量，能不能存在其它等价的，可以方便计算的意义？比如用组合意义转化，或者是用类似 wqs 二分里给式子结合原问题找意义的这种？
    • 有的时候甚至可以考虑增加限制来缩减状态，但是仍然保证合法？类似230115C-T3-实验至少不好算，但是至少凑出还可以维护。
    • 想清楚一类贡献什么时候会被计算，如果它可以被不同种情况的不同位置更新，是否要考虑钦定只计算某些部分：230117C-T4-子图。

    0x03. 思路的延续、切换以及思维空白窗口ψ(｀∇´)ψ

    有性质，有想法之后，应该考虑的就是，怎么样把它应用到原问题上。

    还有，这些东西是否能保证正确性？有没有反例？有没有特殊情况？考虑一下先？

    如果当前思路虽然很清晰，但就是卡在一个点，知道应该把它弄成什么形式，但是确实没有办法做到，就需要考虑重启思路，或者是观察思维链上是不是有什么地方出问题了，比如：230201C-T2-翻转

    思维空白的时候，有两种办法，一种是观察题面，另外一种就是考虑重新理一遍思路。

    长时间的思维空白会导致脱出状态。

    如果感觉思路有点混乱了，或者说卡在关键点但是不知道怎么做了，也需要重新理一下思路，而且不能太快直接过过去，慢一点，不着急。

    0x11. 贪心ψ(｀∇´)ψ

    正常想法其实应该是，对于一看就是贪心，但是策略不明显的题，应该把所有可能策略拿出来，不然反证，不然 exc arg，不然范围放缩，反正就应该求出来什么情况下什么最优，然后再来考虑怎么写代码。

    也就是说，要确保你的结论的正确和合理。

    而不是一个劲的瞎想结论，这样命中效率很低。

    • 一种拆分的贪心思想：可以按照某种“重要程度”分开考虑，然后对于每一种情况分开考虑，最后直接暴力复合：THUPC2023-pre-A - 大富翁
    • 如果一个题，没法很好的通过贪心来快速确定一些必选策略，或者说排除冗杂（也就是有不少特例），此时我们可以考虑直接枚举每一个位置，对他直接通过暴力或者维护信息算答案：CF1801B - Buying gifts。或者是我们就直接考虑 dp：CF1801C-Music Festival

    0x12. 构造ψ(｀∇´)ψ

    其实构造属于比较 adhoc 的题目，不过还是存在一定的套路。

    • 假设答案存在，应该是怎么样的？
    • 那么，答案不存在的时候应该是怎么样？
    • 有没有什么特殊的方式，能够让条件被很好的满足？
    • 可不可以放宽或者缩小题目的限制以便更好的构造？

    0x13. 乱搞ψ(｀∇´)ψ

    • 对于一个序列做很多个 independent problem 的时候一般都会要考虑从上一个状态继承，或者是考虑先整体做一遍再计算 ignore 一些操作之后的影响之类的 ：Least Elements，230203C-T2-排列。
    • 如果是通过打表找规律，找到了 \(k = c\) （常数）的规律，可以考虑把式子中的常数项换成 \(k\) 相关的式子，进而推出通项公式：THUPC2023-pre-B - 拧螺丝
    • 有些时候我们可以考虑忽略部分限制，直接对于一部分限制做完，然后再一一调整满足其它限制，不过在这个过程中通常就可以看出复合后的做法了：Lost Cows

    0x20. 动态规划ψ(｀∇´)ψ

    建议先阅读 关于 dp 的一些注意事项

    0x21. 确定阶段ψ(｀∇´)ψ

    阶段其实很好确定，前 \(i\)，\(u\) 的子树这种是比较常见的，Floyd 的 \(k\) 为阶段算是一种比较特殊的阶段，不过大家都知道了就不写了。

    • 如果题目给出的条件无法以线性的顺序作为阶段，比如 Flexible String Revisit 这样要在序列上随机选择的，不然就考虑每一步的代价（这题是从 \(i\) 个不同变化到 \(i - 1\) 个不同为一步），不然就考虑类似状压拆分这种。

    0x22. 状态设计ψ(｀∇´)ψ

    一般来说只需要提取题目中的要素，结合阶段设计状态就可以。

    常见的 dp 状态无非就是前 \(i\) 前 \(j\)，\(u\) 的子树，状态为 \(msk\)，走到 \(pos\) 的期望这样，看看 /algor/dp 里面的东西就行了。

    不过很多时候限制条件分两类，一类可以在状态中表达，一类可以在转移中表达，这个接下来会说。

    能在状态中表达的一般是当前位置的一些限制，比如当前已经选了 \(j\) 个这种。

    这里记录了一些对我来说曾经不是很常规的想法。

    • 有些题目需要考虑，在状态中钦定前面一段一定合法，然后只考虑决策时会有变化的部分，这种一般在做转移的时候要小心谨慎，不要漏掉了限制条件。
    • 如果题目中的要素比较多，但是知道一定的信息之后可以推出剩下的信息，我们可以在状态里省去对应的维度：Mobile Service
    • 给一个序列，要求将其分成若干个子段，求某个代价的最大值这一类问题，如果题目没有限制你需要分多少段，可以直接设计 dp 状态为，考虑前 \(i\) 个位置分割成若干段的代价最大是多少：Optimal Partition，任务安排

    0x23. 状态转移ψ(｀∇´)ψ

    状态转移一般取决于状态的定义，就考虑怎么让状态能覆盖整个状态空间，不重复不漏算就可以。

    如果状态确实怎么搞都会重复，可以考虑容斥一下。

    如果会漏状态，思考一下是不是阶段有问题，需不需要对原问题做类似 Exchange Argument 的处理？：Zabuton

    或者是不是状态本身就有问题？

    有些限制条件需要在状态转移的时候进行优化，这个就不同于状态设计中的限制条件了，所以在考虑转移的时候，如果没有思路了，可以多看看题面，再找找有没有漏掉的限制条件。

    • 期望的本质实际上是一种加权平均数，所以我们在转移的时候可以把期望当成随机变量的一种取值，可以用来“代表”所有的情况：Ilya and Escalator
    • 换根 dp 的时候需要考虑清楚，第二个 dp 数组到底从什么转移过来，有没有算重复的部分，如果取模还要考虑是否存在逆元之类的：Subtree
    • 有些 dp 虽然阶段是下标，但是它转移的时候，为了决策方便，需要更改一下等号左边的下标，方便转移，不然我们不好确定这个决策到底对答案产生了什么贡献：Cleaning Shifts
    • 如果 dp 状态直接转移很麻烦，比如那种有关联性的状压 dp，考虑一下等价的转化？2301113C-T4-coin
    • 如果是树形 dp，可以利用 dfs 序的性质，把节点分成两类，用于很好的表达一些祖先关系，很多时候还可以用来减少枚举，直接一步到位：230224C-T4-辣椒

    0x24. 后效性相关ψ(｀∇´)ψ

    这个，其实就是方程推出来的，要写代码的时候需要考虑的了。

    • 很多时候需要考虑高斯消元或者更改枚举顺序，如果这样还是没用大概率是状态设计就出事了：[HNOI2013] 游走
    • 期望 dp 很多时候要考虑倒推，不然容易因为状态之间的依赖成环，或者因为终止状态不唯一，甚至因为概率不好计算而导致不必要的麻烦：Gotta Go Fast

    0x25. 优化转移ψ(｀∇´)ψ

    这个其实就分两种，一种是考虑优化转移（决策集合），一种是优化状态。（至少以目前我所知道的来看）

    前者一般是数据结构优化 dp（单调队列也是数据结构啊），也有考虑观察限制条件的特殊性来去除冗杂状态的：Xor-Subsequence (easy version)

    也有利用费用提前计算的思想的：任务安排，这个可以理解成“懒操作”的反方向。

    后者其实一般在状态设计的时候就做了，这里不多提。

    0x26. 各种模型ψ(｀∇´)ψ

    对于一些特定的问题，它们的 dp 设计是非常套路化的，所以单独提出来：

    • 对于本质不同子序列，一般都是设 \(dp(i)\) 考虑前 \(i\) 个位置的答案，然后对于当前位置分类讨论即可：230115C-T4-美食
    • 对于一个排列，算一些方案数的计数 dp，他的固有套路就是考虑一个一个插入：230111C-T4-小W与大数
